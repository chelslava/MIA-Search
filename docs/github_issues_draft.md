# Черновик задач для GitHub Issues (план 0.1.1+)

Рекомендуемые labels: `backend`, `frontend`, `infra`, `qa`, `docs`, `ux`, `p0`, `p1`, `p2`.

## 1. [P0] Стабилизировать CI на Linux для Tauri/GTK зависимостей
- Labels: `infra`, `qa`, `p0`
- Description:
  - Проверить и закрепить установку системных пакетов для `ubuntu-latest`.
  - Убедиться, что `pkg-config` видит `glib-2.0`/`gobject-2.0`.
- Acceptance Criteria:
  - `cargo test` стабильно проходит в Linux job.
  - Нет флапающих падений `glib-sys`/`gobject-sys`.

## 2. [P0] Защитить индексатор от параллельных rebuild
- Labels: `backend`, `p0`
- Description:
  - Добавить защиту от одновременного запуска `index_rebuild`.
  - Возвращать понятный статус при повторном вызове во время активной индексации.
- Acceptance Criteria:
  - Нельзя запустить 2 rebuild одновременно.
  - UI получает корректный и предсказуемый ответ.

## 3. [P0] Усилить регрессионные тесты Scan/Index fallback
- Labels: `backend`, `qa`, `p0`
- Description:
  - Добавить тесты на сценарии пустого индекса и автоматического fallback на scan.
  - Добавить тесты на корректность двухфазной выдачи.
- Acceptance Criteria:
  - Есть тесты на fallback и metadata enrichment.
  - `cargo test` проходит локально и в CI.

## 4. [P1] Улучшить диагностику `index_status` в UI
- Labels: `frontend`, `backend`, `ux`, `p1`
- Description:
  - Показать причину/подсказку, если индекс не готов или rebuild завершился с ошибкой.
  - Добавить рекомендации пользователю в блоке статуса индекса.
- Acceptance Criteria:
  - Для `empty/error/in-progress/ready` есть понятные сообщения.
  - Пользователь понимает, что делать дальше.

## 5. [P1] Довести авто-реиндексацию (startup + TTL) до стабильного поведения
- Labels: `backend`, `p1`
- Description:
  - Уточнить и зафиксировать логику запуска авто-реиндексации при старте.
  - Ввести явные ограничения на частоту проверок.
- Acceptance Criteria:
  - Авто-реиндексация не запускается слишком часто.
  - Поведение предсказуемо при перезапуске приложения.

## 6. [P1] Снизить риск подвисаний UI при интенсивном потоке результатов
- Labels: `frontend`, `ux`, `p1`
- Description:
  - Профилировать обновление списка результатов.
  - Дополнительно ограничить частоту тяжелых обновлений и ререндеров.
- Acceptance Criteria:
  - Нет заметных фризов при больших выборках.
  - UX остается отзывчивым во время поиска.

## 7. [P1] Формализовать release-процесс для тега 0.1.x
- Labels: `infra`, `docs`, `p1`
- Description:
  - Привести release-playbook и smoke-checklist к шаблону версии `0.1.x`.
  - Убедиться, что release notes подтягиваются из `docs/github_release_<tag>.md`.
- Acceptance Criteria:
  - Релиз можно выполнить по одному документу без ручных догадок.
  - Для нового тега используется корректный release body.

## 8. [P2] Подготовить triage-шаблон поддержки (P0/P1/P2 + SLA)
- Labels: `docs`, `qa`, `p2`
- Description:
  - Зафиксировать процесс разбора инцидентов после релиза.
  - Определить ожидаемые сроки реакции для каждого приоритета.
- Acceptance Criteria:
  - Есть документ с матрицей приоритета и SLA.
