# Roadmap с датами

Стартовая дата: **24 марта 2026**  
Формат: 5 спринтов по 2 недели (10 недель суммарно).

## Sprint 1 (24 марта 2026 - 6 апреля 2026)
Цель: каркас приложения и базовое ядро поиска.
- Инициализация Tauri + React + TS.
- Архитектурные слои backend.
- Контракты данных.
- Черновые IPC-команды.
- Базовая интеграция `rust_search`.
Критерий выхода:
- Приложение запускается, выполняет простой поиск по одному root.

## Sprint 2 (7 апреля 2026 - 20 апреля 2026)
Цель: завершить MVP-поисковое поведение.
- Multi-root, strict, ignore_case, hidden, depth, limit.
- Режим files/dirs/mixed через `custom_filter`.
- Потоковая выдача и отмена поиска.
- Базовый UI: search bar, roots panel, results list.
Критерий выхода:
- Неблокирующий потоковый поиск из UI по нескольким корням.

## Sprint 3 (21 апреля 2026 - 4 мая 2026)
Цель: расширенные фильтры и метаданные.
- Фильтры размера и дат.
- Мультирасширение (merge + dedupe).
- Сортировка relevance/name/size/modified/type.
- MetadataService и правая панель деталей.
Критерий выхода:
- Полный набор фильтров и сортировок из ТЗ работает стабильно.

## Sprint 4 (5 мая 2026 - 18 мая 2026)
Цель: эксплуатационная пригодность и UX.
- Settings/history/favorites/presets.
- Статус-бар, темы, чипы фильтров, toasts, skeleton.
- Виртуализация списка.
- Горячие клавиши и контекстные действия.
- Платформенные действия с путями.
Критерий выхода:
- Приложение удобно для повседневного использования.

## Sprint 5 (19 мая 2026 - 1 июня 2026)
Цель: финализация и выпуск.
- Локализация RU/EN (минимально полный UI).
- Unit + integration + UI smoke тесты.
- Диагностические метрики.
- Release-сборки и финальный regression/smoke.
Критерий выхода:
- Готов релиз-кандидат с закрытыми критериями приёмки.

## Milestones
1. **6 апреля 2026**: M1 Каркас и базовый backend.
2. **20 апреля 2026**: M2 MVP-поиск (потоковый, отменяемый).
3. **4 мая 2026**: M3 Фильтры/сортировка/метаданные.
4. **18 мая 2026**: M4 UX + хранилища + действия.
5. **1 июня 2026**: M5 RC и готовность к релизу.

## Риски и буфер
- Риск: платформенные различия в системных действиях (`reveal/open`).
- Риск: производительность на очень больших деревьях каталогов.
- Риск: корректность метаданных на разных ФС.
- Буфер: 10-15% времени каждого спринта оставлять под интеграционные правки.

## План рефакторинга `App.tsx` (старт немедленно)
Цель: разрезать текущий монолитный компонент на независимые модули без регрессии UX и поиска.

### Этап 1. Декомпозиция типов и констант (0.5 дня)
- Вынести темы, форматтеры (`formatBytes`, `formatDate`) и UI-константы в `src/app/config/*` и `src/app/utils/*`.
- Вынести типы локального UI-состояния (`RootItem`, `DisplayMode`, `FilterChip`) в `src/app/types.ts`.
- Критерий: `App.tsx` содержит только импорт и использование, без крупных блоков констант/утилит.

### Этап 2. Выделение бизнес-логики в hooks (1 день)
- Создать `useSearchController` (поиск, отмена, статус, события Tauri, прогресс, elapsed).
- Создать `usePersistenceState` (favorites/history/profiles CRUD).
- Создать `useThemeManager` (встроенные/кастомные темы, localStorage, применение CSS переменных).
- Критерий: в `App.tsx` нет прямой работы с Tauri API и localStorage.

### Этап 3. Выделение состояния фильтров (1 день)
- Создать `useSearchFilters` с `buildCurrentRequest()`, `clearAllFilters()`, фильтр-чипами.
- Перенести логику live search debounce в `useLiveSearch`.
- Критерий: `App.tsx` не содержит сборку `SearchRequest` и преобразования фильтров.

### Этап 4. Разбиение UI на компоненты (1.5 дня)
- `TopToolbar`, `AdvancedFiltersPanel`, `SettingsPanel`.
- `LeftSidebar`, `ResultsWorkspace`, `DetailsSidebar`, `StatusBar`, `AppContextMenu`.
- Критерий: `App.tsx` выполняет роль композиционного контейнера (оркестрация только на уровне props).

### Этап 5. Локализация и i18n-интеграция (1 день)
- Полностью убрать хардкод строк из `App.tsx` и дочерних компонентов.
- Ввести единые namespaces: `app`, `filters`, `status`, `errors`, `actions`.
- Критерий: переключение `ru/en` через `i18next` меняет все видимые строки без перезагрузки.

### Этап 6. Тесты и стабилизация (1 день)
- Обновить smoke-тесты под новую структуру.
- Добавить unit-тесты на `buildCurrentRequest`, фильтр-чипы, преобразование limit/depth/date.
- Критерий: `npm run check`, `npm run test`, `npm run build` зелёные.

### Контрольные метрики рефакторинга
- Целевой размер `App.tsx`: до 250-300 строк.
- Cyclomatic complexity в `App.tsx`: минимальная (без длинных ветвлений и крупных `useEffect`).
- Все сайд-эффекты изолированы в hooks.
- UI-компоненты покрыты smoke-тестом и не зависят напрямую от Tauri-клиента.
